### 프로젝트 작성 가이드

* @QueryProjection DTO 사용 기준

    - 검색 결과를 응답에 사용할 때

    - 검색 결과가 많을 시 최적화가 필요할 때

    - 단순 중복 체크 등에는 사용 x

        - 엔티티로 반환해도 됨

* 응답 객체/조회 전용 객체 분리

    - 클라이언트의 응답에 사용하는 객체와 조회 결과를 매핑하는 DTO 는(쿼리 프로젝션 등) 분리해야 함

    - 분리하지 않는 경우 발생하는 일

        - 응답 요구사항이 바뀌면 Reposioty 계층에도 변화가 생긴다

        - Reposioty 계층에서 클라이언트 응답 객체를 사용하고 있었기 때문

        - 즉 계층간 결합도가 높아지고 따라서 유지보수가 어려워진다

    - 응답 전용 객체는 사용자 응답에만 사용
    
    - 조회 전용 DTO 를 사용했다면 다시 응답 전용 객체로 변환해 사용자에게 응답해야 한다

* DTO/Entity 계층 분리

    - DTO 내부에서 Entity 타입을 직접 사용하거나, Entity를 생성자/필드로 두지 말아야 함

    - DTO 와 Entity 는 책임이 다르다

        - DTO : 요청/응답 데이터에 맞춰 설계되는 객체

        - Entity : DB 와 매핑되는 객체

    - DTO 가 엔티티를 의존하는 구조 사용 X

        ```java
        @Entity
        public class UserEntity {

            private String email;
            private String name;
        }

        @Getter
        public class UserResponse {

            private final String email;
            private final String name;

            // 필드가 엔티티 타입을 가지는 경우
            private final UserEntity entity; 

            // 생성자의 인자로 엔티티 받는 경우
            public UserResponse(UserEntity entity) {
                this.email = entity.getEmail();
                this.name = entity.getName();
            }

            // @QueryProjection DTO 가 엔티티를 통째로 받는 경우
            @QueryProjection
            public UserResponse(UserEntity entity) {
                this.email = entity.getEmail();
                this.name = entity.getName();
            }
        }
        ```

        - api 디자인이 변경되면 엔티티도 바꿔야 함

            - 프론트에서 name 대신 firstName, lastName 으로 변경 요청

            ```java
            public class UserResponse {
                private final String email;

                // 필드 이름 변경
                private final String firstlName;

                // 필드 추가
                private final String lastName;

                // 문제 발생
                public UserResponse(UserEntity entity) {
                    this.email = entity.getEmail();
                    this.name = ???;    // 만들 수 없음
                }
            }
            ```

            - UserEntity 는 name 필드만 있고 fullName, lastName 이 없음

            - 그렇다고 Entity 에 fullName, lastName 필드를 추가하면 광범위하게 영향이 감

                - DB 수정

                - 테스트 깨짐

                - 수정된 Entity 를 사용하는 다른 클래스들 영향

            - 즉 API 가 엔티티를 끌고 움직이는 계층 오염이 발생됨

        - DB 변경 시 DTO 도 변경해야 함

            ```java
            // Entity 수정
            private String nickname;

            // DTO
            public UserResponse(UserEntity entity) {
                this.name = entity.getName();  // 컴파일 오류
            }
            ```

* Entity 변경 기준

    - 도메인 규칙이 바뀌었을 때

        - 회원 관리에 닉네임 이라는 개념이 새로 생겼을 경우

        - DB, Entity 둘 다 변경해야 함

    - API 표현을 위한 가공 정보가 바뀌었을 때

        - 변경하면 안됨

        - 엔티티는 그대로 두고 맵퍼나 DTO 에서 조합해야 함

* exists 패턴 사용

    ```java
    public boolean existsByEmail(String email) {

    return queryFactory
            .selectOne()
            .from(userEntity)
            .where(userEntity.email.eq(email))
            .fetchFirst() != null;
    }
    ```

    - 조건을 만족하는 row 를 찾으면 바로 결과를 반환하고 쿼리를 종료

    - 응답 데이터가 필요 없는 단순 중복 체크, 존재 확인에 사용
    
        - 쿼리 프로젝션을 사용할 필요도 없음 

    - 주의사항 : 조회결과가 없으면 null 이 반환되기 때문에 null 로 체크해야 함

* 예외 응답 구조

    - 비즈니스 전용 예외 코드 전용 Enum 사용

    - 비즈니스 전용 예외 클래스 사용

        - 비즈니스 예외 시 해당 예외 클래스 생성

    - 공통 예외 응답 객체 사용

    - 컨트롤러 어드바이스에서 비즈니스 예외, 시스템 예외 등을 받아 처리

        - 예외 처리 시 비즈니스 예외 코드 Enum 으로 공통 예외 응답 객체 생성

        - http 상태 코드를 지정하고 바디에 예외 응답 객체를 사용해 응답 

* 로깅 사용 기준 정하기

    - 어디서 어떤 레벨로 로깅할지 규칙을 정해야 한다

    - 예시

        - 컨트롤러

            ```java
            @PostMapping("/users")
            public UserResponse signUp(@RequestBody UserRequest request) {
                log.info("[UserSignUp] start email={}", request.getEmail());
                UserResponse response = userService.signUp(request);
                log.info("[UserSignUp] success userId={}", response.getId());
                return response;
            }
            ```

            - INFO : 요청 시작/종료에 로깅하며 핵심 요약 기록

            - DEBUG : 요청 바디, 세부 파라미터 (개발 환경)

        - 서비스

            ```java
            @Transactional
            public UserResponse signUp(UserRequest request) {

                if (userRepository.existsByEmail(request.getEmail())) {
                    log.warn("[UserSignUp] duplicated email={}", request.getEmail());
                    throw new UserException("EMAIL_DUPLICATED");
                }

                User user = userRepository.save(UserMapper.toEntity(request));
                log.debug("[UserSignUp] new user saved id={}", user.getId());

                return UserMapper.toResponse(user);
            }
            ```

            - INFO : 비즈니스 흐름의 중요한 이벤트

            - DEBUG : 디버그가 필요할 만한 내부 상태

        - 예외 처리

            ```java
            @Slf4j
            @RestControllerAdvice
            public class GlobalExceptionHandler {

                @ExceptionHandler(UserException.class)
                public ResponseEntity<ErrorResponse> handleUserException(UserException e) {
                    log.warn("[UserException] code={}, message={}", e.getCode(), e.getMessage());
                    return ResponseEntity
                            .status(HttpStatus.BAD_REQUEST)
                            .body(new ErrorResponse(e.getCode(), e.getMessage()));
                }

                @ExceptionHandler(Exception.class)
                public ResponseEntity<ErrorResponse> handleException(Exception e) {
                    log.error("[UnexpectedException]", e); // 스택트레이스 출력
                    return ResponseEntity
                            .status(HttpStatus.INTERNAL_SERVER_ERROR)
                            .body(new ErrorResponse("INTERNAL_SERVER_ERROR", "잠시 후 다시 시도해주세요."));
                }
            }
            ```

            - WARN : 예상 가능한 예외 (클라이언트 예외 등)

            - ERROR : 예상하지 못한 예외
