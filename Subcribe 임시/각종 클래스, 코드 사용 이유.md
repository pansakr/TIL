### JPA

* Long 사용 이유

    - Postgreql 의 integer 타입은 java 의 int/integer, bigint 타입은 long/Long 타입과 매핑됨

    - DB가 integer 여도 long/Long 로 매핑해도 상관 없음

        - integer(32비트) 범위를 long(64)비트가 포함하기 때문

    - DB 타입과 상관없이 Wrapper 클래스인 Long 타입을 사용하는게 권장됨

        - jpa 의 GenerationType.IDENTITY 옵션은 Long 타입을 전제로 동작하는 경우가 많음

        - 다른 타입이라면 오류가 발생할 할 수도 있음

* auditior

    - 엔티티의 생성/수정 시점과 작성자/수정자를 자동으로 기록하기 위한 기능

    - @EnableJpaAuditing
    
        - 엔티티의 생성일자/수정일자(@CreatedDate, @LastModifiedDate), 작성자/수정자(@CreatedBy, @LastModifiedby) 필드에 값을 자동으로 채워주는 기능을 활성화

        - 이 엔티티가 없다면 @CreatedDate, @LastModifiedDate, @CreatedBy, @LastModifiedby 가 작동하지 않음

    - AuditorAware<T>

        - 엔티티의 작성자/수정자(@CreatedBy, @LastModifiedby) 필드에 들어갈 값을 제공

        - 이 설정이 없다면 현재 사용자가 누구인지 알 수 없음

        - AuditorAware 를 구현한 다음 설정 파일에 @Bean 으로 등록해서 사용

        - 생성일자/수정일자 는 애플리케이션이 현재 시간을 자동으로 입력하므로 별도 설정 필요 없음

        ```java
        public class DefaultAuditorAware implements AuditorAware<String> {
            
            /*시큐리티 미사용 시*/ 
            @Override
            public Optional<String> getCurrentAuditor() {
                
                String username = "system";
                
                return Optional.of(username);
            }
        }
        ```

    - @EntityListeners(AuditingEntityListener.class)

        - 엔티티의 저장/수정을 이벤트를 감지해서 @EnableJpaAuditing, AuditorAware<T> 를 사용해 자동으로 필드에 값을 채움

        - 리스너가 없다면 언제 저장/수정 되는지 알 수 없기 때문에 위의 두 설정이 있더라도 자동 기록을 할 수 없음

        ```java
        @EntityListeners(value = { AuditingEntityListener.class })
        @MappedSuperclass
        public class BaseEntity {

            @CreatedDate
            @Column(name = "created_at")
            private LocalDateTime createdDate;

            @LastModifiedDate
            @Column(name = "updated_at")
            private LocalDateTime updatedDate;
        }
        ```
